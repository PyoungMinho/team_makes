# 프로젝트 방향서 (확정)

> 팀장 합동 회의 결과 / 2026-04-11 확정
> 참석: 기획팀장(사회), 디자인팀장, 프론트팀장, 백엔드팀장

---

## 1. 프로젝트 개요

**반려동물 의료기록 전자수첩 + 응급 트리아지 챗봇을 핵심으로 하는 보호자 안심 플랫폼 (iOS 우선, Phase 1에서 Android 확장)**

---

## 2. 확정된 기술 스택

### 프론트엔드

| 항목 | 결정 | 근거 |
|---|---|---|
| 프레임워크 | **React Native (Expo)** | iOS-Android 코드 공유 90%, Phase 1 Android 확장에 최적 |
| 상태관리 | Zustand + React Query | 경량 + 서버 상태 캐싱 분리 |
| 챗봇 통신 | SSE (Server-Sent Events) | 백엔드 SSE 스트리밍과 1:1 대응, 저사양 기기 고려 |
| 로컬 저장 | AsyncStorage + SQLite | 오프라인 의료기록 조회 대응 |

### 백엔드

| 항목 | 결정 | 근거 |
|---|---|---|
| 아키텍처 | **Modular Monolith** (Phase 0-1) | 초기 운영 복잡도 최소화, Phase 2에서 챗봇/보험 모듈 분리 |
| 프레임워크 | **NestJS (TypeScript)** | 모듈 구조가 Modular Monolith에 적합, 프론트와 언어 통일 |
| DB | **PostgreSQL + Prisma ORM** | 의료기록의 관계형 데이터에 최적, Prisma로 타입 안전성 확보 |
| 캐시 | **Redis** | 세션, 챗봇 대화 컨텍스트, 빈번 조회 데이터 캐싱 |
| 파일 저장 | **AWS S3** | 영수증 이미지, 의료 문서 저장 |
| API 방식 | **REST 기본 + 챗봇 SSE 스트리밍** | REST로 CRUD 처리, 챗봇만 SSE로 실시간 스트리밍 |

### 인프라 (Phase 0-1 기준)

| 항목 | 결정 |
|---|---|
| 클라우드 | AWS (ap-northeast-2) |
| 컨테이너 | Docker + ECS Fargate |
| CI/CD | GitHub Actions |
| 모니터링 | CloudWatch + Sentry |
| OCR | AWS Textract 또는 Naver Clova OCR (한국어 영수증 정확도 비교 후 확정) |
| LLM | OpenAI GPT-4o-mini (비용 효율) / 응급도 높은 쿼리만 GPT-4o 라우팅 |

## 3. 확정된 화면 구조

### 하단 3탭 구조 (디자인팀장 제안 채택)

```
[내 아이]          [응급]           [더보기]
  메인 탭           챗봇 탭          설정/부가 탭
```

### 탭별 핵심 화면

**탭 1: 내 아이**
- 펫 프로필 목록 (멀티펫 대응, 카드 스와이프)
- 펫 상세: 기본정보 + 의료기록 타임라인
- 의료기록 추가: 수동입력 + 영수증 OCR 촬영
- 예방접종 일정 캘린더 (Phase 1)

**탭 2: 응급**
- 챗봇 대화 화면 (SSE 스트리밍, 타이핑 인디케이터)
- 증상 빠른선택 카드 (구토, 설사, 외상 등 8개 카테고리)
- 트리아지 결과 + 근처 야간 병원 지도
- 면책 고지 상시 노출 배너

**탭 3: 더보기**
- 보호자 프로필 / 계정 설정
- 보험 청구 서류 생성 (Phase 1)
- 알림 설정
- 앱 정보 / 이용약관

### 추가 필수 화면
- 온보딩 플로우 (3-4스텝: 보호자 정보 -> 첫 펫 등록 -> 핵심 기능 안내 -> 권한 요청)
- 에러/빈 상태 화면 (네트워크 오류, 데이터 없음, 서버 점검)
- 앱스토어 리젝 대비 면책 UI (앱 첫 실행 시 동의 화면)

## 4. 기획서 보완 사항 (즉시 결정 필요)

3개 팀장이 공통적으로 지적한 기획서 미비 사항을 정리하고, 각 항목에 대해 본 회의에서 방향을 확정한다.

### 4-1. 즉시 확정 사항

| 항목 | 결정 | 결정 근거 |
|---|---|---|
| **의료기록 입력 방식** | 수동입력 + 영수증 OCR **병행** | 디자인/백엔드 팀장 공통 의견. 수동입력만으로는 이탈률 치명적. OCR은 MVP 필수 기능으로 격상 |
| **멀티펫 구조** | 처음부터 1:N(보호자:펫) 설계 | 보호자 30%가 2마리 이상. DB 스키마, UI 모두 Day 1부터 멀티펫 전제 |
| **공동보호자** | Phase 0에서는 **미지원**, Phase 1에서 도입 검토 | 아키텍처 복잡도 증가 vs 초기 필요성 대비. 단, DB 스키마는 확장 가능하게 설계 |
| **과금 모델** | Phase 0 완전 무료, Phase 1에서 프리미엄 구독 테스트 | 원안 유지. 무료 티어: 월 챗봇 3회 + 기본 기록 / 프리미엄: 월 3,900원 |
| **오프라인 시나리오** | 의료기록 조회는 로컬 캐싱으로 오프라인 지원, 챗봇은 온라인 필수 | 응급 상황에서 기존 기록 확인은 필수이므로 로컬 캐싱 적용 |

### 4-2. 기획팀 추가 산출물 (W1 내 완료 필수)

3개 팀장이 공통적으로 요청한 기획 산출물을 정리한다. 이 산출물이 없으면 디자인/개발 착수 불가.

| 산출물 | 내용 | 소비 팀 | 마감 |
|---|---|---|---|
| **의료기록 입력필드 정의서** | 필드명, 타입, 필수/선택, 검증규칙, OCR 매핑 | 디자인, 프론트, 백엔드 | W1 |
| **챗봇 시나리오 문서** | 최소 3개 시나리오 (구토/설사, 외상/출혈, 호흡곤란) + 응답 포맷 명세 + 폴백 시나리오 | 디자인, 프론트, 백엔드 | W1 |
| **온보딩 플로우 정의** | 스텝별 화면, 수집 정보, 스킵 가능 여부 | 디자인 | W1 |
| **에러 상태 목록** | 화면별 에러 케이스 + 사용자 안내 문구 | 디자인, 프론트 | W2 |
| **브랜딩 방향 키워드** | 톤앤매너, 컬러 방향, 일러스트 스타일 키워드 | 디자인 | W1 |

### 핵심 마일스톤

| 시점 | 마일스톤 | 게이트 조건 |

| **W4 시작** | 프론트 개발 착수 | 와이어프레임 v1 리뷰 완료 |
| **W4** | 중간 리뷰 (전팀) | 와이어프레임 + API 스펙 초안 동기화 |

| **W11** | 앱스토어 제출 | QA 완료 + 면책 UI 검증 |
| **W13** | 클로즈드 베타 시작 | 앱스토어 승인 + 파트너 병원 10곳 MOU 완료 |
| **W16** | Go/No-Go 판단 | 주간 활성 40%, 의료기록 월 500건, 챗봇 NPS 30 |

### 의존성 경고

- 프론트 W4 착수는 디자인 와이어프레임 v1 완료에 **절대 의존**. 디자인 지연 시 프론트 일정 전체 밀림.
- 백엔드 OCR PoC(W3)가 실패하면 대안 OCR 서비스 전환에 1-2주 추가 소요 가능.
- 앱스토어 심사 리젝 시 W11-W12 버퍼 소진. 3회 이상 리젝 시 일정 재조정 필요.

## 6. 리스크 Top 5 + 대응책

| 순위 | 리스크 | 영향도 | 발생확률 | 대응책 |
|---|---|---|---|---|
| **1** | **의료기록 입력 허들로 인한 사용자 이탈** | 치명 | 높음 | 영수증 OCR을 MVP 필수 기능으로 격상. 촬영 한 번으로 핵심 필드 자동 입력. 수동입력은 OCR 실패 시 폴백으로만 사용. 온보딩에서 OCR 체험 유도. |
| **2** | **챗봇 스트리밍 성능 (저사양 기기)** | 높음 | 중간 | SSE 청크 사이즈 최적화, 응답 토큰 제한(최대 500토큰), 저사양 기기 감지 시 스트리밍 비활성화 + 완성 응답 전송 폴백. W5-W6에서 기기별 성능 테스트 필수. |
| **3** | **앱스토어 의료 관련 심사 리젝** | 높음 | 중간 | 면책 UI를 앱 첫 실행 + 챗봇 진입 시 이중 배치. "수의사 판단을 대체하지 않는 참고 정보" 문구 법률 검토 완료 후 적용. 심사 가이드라인 사전 분석. 리젝 대비 2주 버퍼 확보 완료. |
| **4** | **LLM API 비용 폭증** | 중간 | 중간 | 2단계 라우팅: 단순 질문은 GPT-4o-mini($0.15/1M input), 고위험 판단만 GPT-4o. 응답 캐싱(Redis, 동일 증상 조합 24h TTL). 사용자별 일일 쿼리 제한. MAU 5만 기준 월 $10,000 상한 설정, 초과 시 알림. |
| **5** | **디자인 일정 지연 -> 프론트 착수 지연** | 높음 | 중간 | 디자인-프론트 주 2회 싱크 미팅. 와이어프레임 확정 전이라도 공통 컴포넌트(버튼, 입력, 카드)는 프론트 선행 개발 가능. W3 와이어프레임 미완료 시 W4 긴급 리뷰 소집. |
