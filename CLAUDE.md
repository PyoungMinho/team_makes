# Team Makes - AI Agent Company

## 프로젝트 개요
AI 에이전트 기반의 가상 팀(회사) 시스템. 각 에이전트가 직원으로서 고유한 역할과 페르소나를 가지며, deskRPG 2D 가상 오피스에서 시각화된다.

## 팀 구조 (총 16명)

| 팀 | 인원 | 구성 | 핵심 MCP |
|---|---|---|---|
| 기획팀 | 5명 | 팀장(Opus) + 창의(Sonnet+Haiku) + 비평(Sonnet+Haiku) | WebSearch, Sheets, Figma |
| 디자인팀 | 3명 | 팀장(Opus) + UX(Sonnet) + UI(Sonnet) | Figma |
| 프론트엔드팀 | 3명 | 팀장(Opus) + 컴포넌트(Sonnet) + 페이지(Sonnet) | Figma |
| 백엔드팀 | 3명 | 팀장(Opus) + API(Sonnet) + DB(Sonnet) | Figma |
| QA팀 | 2명 | QA설계자(Opus) + QA실행자(Opus) | - |

## 기획팀 회의 시스템 (5인)

기획팀은 **회의 토론 방식**으로 운영된다. 사용자가 주제를 던지면 3라운드로 진행:

```
ROUND 1 — 브레인스토밍 (Haiku x2, 동시 실행)
  @창의팀원 → 아이디어 5개 이상 빠르게 발산
  @비평팀원 → 문제점 3개 이상 빠르게 지적

ROUND 2 — 종합 (Sonnet x2, 동시 실행)
  @창의기획자 → 창의팀원 아이디어 종합 + 보강 + 제안서
  @비평기획자 → 비평팀원 의견 종합 + 대안 제시 + 분석서

ROUND 3 — 결정 (Opus x1)
  @기획팀장 → 양측 종합, 최종 결론 + 산출물 작성
```

### 기획팀 멤버

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| @기획팀장 | Opus | 최종 결정권자 |
| @창의기획자 | Sonnet | 창의 파트 리더, 아이디어 종합 |
| @창의팀원 | Haiku | 아이디어 빠른 발산 |
| @비평기획자 | Sonnet | 비평 파트 리더, 리스크 분석 |
| @비평팀원 | Haiku | 문제점 빠른 지적 |

## 디자인팀 (3인)

```
@UX디자이너 → 사용자 플로우 + 와이어프레임
@UI디자이너 → 비주얼 스타일 + 디자인 토큰   (동시)
    ↓
@디자인팀장 → 종합하여 최종 디자인 스펙 확정
```

## 프론트엔드팀 (3인)

```
@컴포넌트개발자 → UI 컴포넌트 구현
@페이지개발자 → 페이지 조립 + API 연동   (동시)
    ↓
@프론트팀장 → 코드 리뷰 + 아키텍처 확정
```

## 백엔드팀 (3인)

```
@API설계자 → 엔드포인트 설계 + 인증 흐름
@DB설계자 → 데이터 모델링 + 스키마 설계   (동시)
    ↓
@백엔드팀장 → 시스템 아키텍처 종합 + 보안 확정
```

## QA팀 (2인)

```
@QA설계자 → 테스트 전략 + 케이스 설계 + 코드 리뷰
    ↓
@QA실행자 → 테스트 코드 작성 + 실행 + 버그 리포트
```

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| @QA설계자 | Opus | 테스트 전략, 케이스 설계, 품질 기준, 코드 리뷰 |
| @QA실행자 | Opus | 테스트 구현, 실행, 버그 헌팅, 수정 검증 |

## 에이전트 호출 규칙

- 에이전트는 `@에이전트명`으로 호출
- 각 에이전트는 자신의 전문 분야에서만 작업
- 팀 간 핸드오프는 명시적으로 진행
- 기획팀은 반드시 **3라운드 회의 프로세스**를 거쳐 결론 도출
- 디자인/프론트/백엔드팀은 **팀원 동시 작업 → 팀장 종합** 구조

## 자동 오케스트레이션 (트리거 명령어)

사용자가 아래 명령어를 입력하면, 해당 워크플로우를 **자동으로 전체 실행**한다.
각 라운드의 에이전트를 순서대로 호출하고, 이전 라운드 결과를 다음 라운드에 전달한다.
산출물은 `docs/` 하위에 자동 저장한다.

### "기획 회의" — 기획팀 5인 회의

**트리거**: 사용자가 "기획 회의", "기획팀 회의", "기획해줘" + 주제를 입력
**실행 흐름**:
1. ROUND 1: `@창의팀원`(Haiku) + `@비평팀원`(Haiku) **동시 실행** (Agent 병렬 호출)
   - 창의팀원: 아이디어 7개+ 발산 + 웹 검색
   - 비평팀원: 문제점 5개+ 지적 + 실패 사례 검색
2. ROUND 2: `@창의기획자`(Sonnet) + `@비평기획자`(Sonnet) **동시 실행**
   - 창의기획자: R1 창의팀원 결과를 받아 → 핵심 아이디어 선별 + 제안서
   - 비평기획자: R1 비평팀원 결과를 받아 → 리스크 종합 + 대안 제시
3. ROUND 3: `@기획팀장`(Opus)
   - R2 양측 결과를 모두 받아 → 최종 기획서 작성 → `docs/planning/` 저장

### "팀장 회의" — 4개 팀장 합동 회의

**트리거**: 사용자가 "팀장 회의", "팀장 합동 회의", "팀장들 모여"
**전제**: `docs/planning/` 에 기획서가 존재해야 함
**실행 흐름**:
1. 기획서를 읽어서 핵심 내용 파악
2. `@디자인팀장` + `@프론트팀장` + `@백엔드팀장` **동시 실행** (각자 관점에서 검토)
   - 디자인팀장: UX 관점 + 화면 구조 + 디자인 일정
   - 프론트팀장: 기술 스택 + 구현 난이도 + 프론트 일정
   - 백엔드팀장: 시스템 아키텍처 + API/DB 방향 + 보안
3. `@기획팀장`(Opus): 3개 팀장 의견 종합 → 프로젝트 방향서 확정 → `docs/planning/project-direction.md` 저장

### "디자인 시작" — 디자인팀 3인 작업

**트리거**: 사용자가 "디자인 시작", "디자인팀 작업", "디자인해줘"
**전제**: 프로젝트 방향서가 존재해야 함
**실행 흐름**:
1. `@UX디자이너` + `@UI디자이너` **동시 실행**
   - UX디자이너: 사용자 플로우 + 와이어프레임 + IA → `docs/design/ux-spec.md`
   - UI디자이너: 비주얼 시스템 + 컴포넌트 스펙 + 디자인 토큰 → `docs/design/ui-spec.md`
2. `@디자인팀장`(Opus): 양측 종합 → 최종 디자인 스펙 확정 → `docs/design/design-final.md`

### "개발 시작" — 프론트+백엔드 병렬 개발

**트리거**: 사용자가 "개발 시작", "개발팀 작업", "프론트백엔드 시작"
**전제**: 디자인 스펙이 존재해야 함
**실행 흐름**:
1. 프론트팀 + 백엔드팀 **동시 실행**
   - 프론트: `@컴포넌트개발자` + `@페이지개발자` 동시 → `@프론트팀장` 종합
   - 백엔드: `@API설계자` + `@DB설계자` 동시 → `@백엔드팀장` 종합
2. 산출물: `src/` (프론트 코드) + `api/` (백엔드 코드)

### "QA 돌려" — QA팀 2인 테스트

**트리거**: 사용자가 "QA 돌려", "테스트 해줘", "QA 시작", "버그 찾아줘"
**전제**: 개발된 코드가 존재해야 함
**실행 흐름**:
1. `@QA설계자`(Opus): 기획서 + 코드를 읽고 → 테스트 전략 + 케이스 설계 + 코드 리뷰
2. `@QA실행자`(Opus): QA설계자의 케이스를 받아 → 테스트 코드 작성 + 실행 + 버그 리포트
3. 산출물: `docs/qa/` (테스트 설계서 + 버그 리포트) + 테스트 코드 파일

### "최종 리뷰" — 팀장 합동 리뷰

**트리거**: 사용자가 "최종 리뷰", "전체 리뷰", "팀장 리뷰"
**실행 흐름**:
1. `@디자인팀장` + `@프론트팀장` + `@백엔드팀장` **동시** — 각자 파트 검수
2. `@기획팀장` — 기획 의도 대비 결과 검증 + 최종 리뷰 보고서

### "전체 파이프라인" — STEP 1~5 전부 실행

**트리거**: 사용자가 "전체 파이프라인", "처음부터 끝까지", "풀 파이프라인" + 주제
**실행 흐름**: 기획 회의 → 팀장 회의 → 디자인 시작 → 개발 시작 → 최종 리뷰 순차 실행

---

### 오케스트레이션 규칙

1. **이전 STEP 산출물이 없으면 실행 거부** — "디자인 시작"을 했는데 기획서가 없으면 "기획 회의부터 해주세요" 안내
2. **각 라운드 결과는 반드시 다음 라운드 프롬프트에 포함** — 이전 결과를 읽어서 전달
3. **동시 실행 가능한 에이전트는 반드시 Agent 병렬 호출** — 순차 실행 금지
4. **산출물 자동 저장** — `docs/{팀명}/` 하위에 날짜 없이 최신본 덮어쓰기
5. **진행 상황은 TodoWrite로 실시간 추적** — 각 라운드 시작/완료 표시
6. **에이전트 모델 매핑 준수** — Haiku는 model:haiku, Sonnet은 model:sonnet, Opus는 model:opus로 Agent 호출

## 전체 워크플로우

```
STEP 1. "기획 회의" → 기획서 도출
    ↓
STEP 2. "팀장 회의" → 프로젝트 방향 확정
    ↓
STEP 3. "디자인 시작" → 디자인 스펙
    ↓
STEP 4. "개발 시작" → 프론트+백엔드 병렬 개발
    ↓
STEP 5. "QA 돌려" → 테스트 설계 + 실행 + 버그 리포트
    ↓
STEP 6. "최종 리뷰" → 최종 검수
```

## deskRPG 연동

- 각 에이전트는 deskRPG NPC로 등록
- OpenClaw 게이트웨이를 통해 AI ↔ NPC 바인딩
- 태스크 할당/보고가 가상 오피스 내에서 시각화
